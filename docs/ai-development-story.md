# 我用 AI 开发了一个 K8s 运维工具，全程只花了 30 分钟

## 🤔 起因

作为一个可观测性领域的解决方案专家，我每天都在和日志、监控、告警打交道。

K8s 日志排查是个痛点，尤其是新人接手时，面对几百行日志经常一脸懵。

**我就想：能不能做一个工具，自动识别常见错误，给出修复建议？**

但是... 我最近工作太忙了，没时间写代码。

直到我发现了 AI 编程助手。

## 🚀 开发过程

### 第 1 步：需求描述（2 分钟）

我对 AI 说：

> "帮我做一个 K8s 日志诊断工具，能自动识别常见错误模式，比如 OOMKilled、CrashLoopBackOff，给出修复建议。支持文件输入和 kubectl 输入。"

### 第 2 步：AI 生成代码（5 分钟）

AI 直接给了我一个完整的 Python 脚本：

- 定义了 10+ 种错误模式的正则表达式
- 实现了日志分析和匹配逻辑
- 支持多种输入方式（文件、kubectl、标准输入）
- 提供了文本和 JSON 两种输出格式

**代码质量出乎意料地好**，有类型注解、有文档字符串、有异常处理。

### 第 3 步：测试和调试（10 分钟）

我跑了一下，发现一个小 bug：

```python
output.append()  # 错误：缺少参数
```

AI 立刻修复：

```python
output.append("")  # 正确
```

然后我让 AI 生成了一个示例日志文件，测试效果：

```bash
$ k8s-log-doctor -f example.log

发现 6 个问题:

🔴 [1] OOMKilled
   严重程度: CRITICAL
   问题描述: 容器因内存不足被杀死
   置信度: 95%

   💡 建议:
      1. 增加Pod的memory limit
      2. 检查应用是否有内存泄漏
      ...
```

**完美！** 5 秒钟出结果，识别出 6 个问题。

### 第 4 步：完善文档（5 分钟）

AI 帮我写了：

- README.md - 完整的使用文档
- setup.py - 打包配置
- 示例日志文件
- 使用示例

### 第 5 步：发布到 GitHub（8 分钟）

```bash
git init
git add -A
git commit -m "Initial release: K8s Log Doctor v0.1.0"
git push -u origin main
git tag v0.1.0
```

**全程 30 分钟，工具上线。**

## 🎯 这个工具能做什么？

**K8s Log Doctor** 可以：

- 自动识别 10+ 种常见 K8s 错误
- 按严重程度分级（CRITICAL/HIGH/MEDIUM/LOW）
- 给出具体的修复建议
- 支持文件、kubectl、管道多种输入

**GitHub 地址**: https://github.com/Richardoris/k8s-log-doctor

## 🤔 AI 编程的局限

当然，AI 也不是万能的：

1. **需要明确的需求** - 你得知道要做什么
2. **需要测试验证** - AI 生成的代码不一定 100% 正确
3. **需要领域知识** - 比如 K8s 的错误模式，AI 不知道，我得告诉它
4. **复杂逻辑还是要人工** - 简单的 CRUD 可以，复杂的业务逻辑还得自己来

**AI 是工具，不是替代品。**
